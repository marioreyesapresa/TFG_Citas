RESUMENEsta memoria recoge el contexto, el diseño, la implementación y los resultados obtenidos del Trabajo de Fin de Grado titulado “Diseño e Implementación de un Sistema Inteligente de Reasignación de Citas Médicas”.El proyecto aborda la problemática de la ineficiencia generada por las cancelaciones de última hora en centros sanitarios, que resultan en la pérdida de franjas asistenciales y un aumento de las listas de espera. La solución propuesta consiste en el desarrollo de una plataforma web cuyo núcleo es un motor de reasignación proactiva. Este sistema identifica de forma autónoma a los candidatos idóneos para ocupar huecos liberados mediante un algoritmo de scoring paramétrico que pondera la urgencia clínica, la antigüedad y las preferencias del paciente.Desde el punto de vista técnico, se han implementado mecanismos de seguridad avanzada como el patrón Circuit Breaker para garantizar la estabilidad del servidor, y un módulo de validación documental mediante códigos QR para asegurar la integridad de las recetas emitidas. Los resultados obtenidos validan la viabilidad del sistema, demostrando una optimización significativa en la gestión de agendas y una mejora en la experiencia del usuario mediante un flujo de aceptación automatizado y sin fricciones.El prototipo funcional derivado de este estudio se encuentra desplegado para su evaluación en: https://tfgmarreyapr.pythonanywhere.comEl código fuente del proyecto y el historial de desarrollo están disponibles en el repositorio público: https://github.com/marioreyesapresa/TFG_Citas.gitPalabras clave: Gestión de citas médicas; Motor de reasignación proactivo; Algoritmo de scoring; Validación QR; Circuit Breaker; Optimización de agendas; Ingeniería del Software.

ABSTRACTThis report includes the context, design, implementation, and results obtained from the Final Degree Project entitled “Design and Implementation of an Intelligent System for the Reassignment of Medical Appointments”.The project addresses the problem of inefficiency generated by last-minute cancellations in health centers, which result in the loss of healthcare slots and an increase in waiting lists. The proposed solution consists of the development of a web platform whose core is a proactive reassignment engine. This system autonomously identifies the ideal candidates to fill the vacant slots by means of a parametric scoring algorithm that weights clinical urgency, seniority, and patient preferences.From a technical point of view, advanced security mechanisms have been implemented, such as the Circuit Breaker pattern to guarantee server stability, and a document validation module using QR codes to ensure the integrity of the prescriptions issued. The results obtained validate the viability of the system, demonstrating a significant optimization in agenda management and an improvement in user experience through an automated and frictionless acceptance flow.The functional prototype derived from this study is deployed for evaluation at: https://tfgmarreyapr.pythonanywhere.com.The source code and development history are available in the following public repository: https://github.com/marioreyesapresa/TFG_Citas.git.Keywords: Medical appointment management; Proactive reassignment engine; Scoring algorithm; QR validation; Circuit Breaker; Schedule optimization; Software Engineering.

1. Introducción, Contexto y Motivación En esta sección se presenta la introducción, el contexto y los antecedentes del proyecto para, finalmente, exponer la motivación que ha impulsado el desarrollo de este.1.1 Introducción y contexto El desarrollo de este Trabajo Fin de Grado se enmarca en la necesidad actual de modernizar las herramientas de gestión en el sector de la salud. A diferencia de los sistemas tradicionales, que suelen ser simples repositorios de datos o agendas estáticas, el objetivo de este proyecto es crear una solución proactiva que gestione de forma inteligente el flujo de pacientes.El contexto técnico se centra en el desarrollo de una plataforma web diseñada para solucionar una ineficiencia crítica: la pérdida de capacidad asistencial ante cancelaciones imprevistas. Actualmente, cuando un paciente cancela una cita, el sistema queda en un estado pasivo; el hueco permanece vacío a menos que se realice una gestión manual. Este proyecto introduce un motor de reasignación automatizado que detecta estos eventos y actúa de inmediato para rellenar la agenda basándose en criterios lógicos y técnicos predefinidos.La arquitectura de la aplicación no solo contempla la funcionalidad de usuario, sino que pone el foco en la robustez técnica. Para ello, se han integrado patrones de diseño como el Circuit Breaker (Nygard, 2018), esencial en sistemas distribuidos para evitar fallos en cascada, y un sistema de validación documental mediante códigos QR, que aporta una capa extra de seguridad y verificación externa. Por tanto, el trabajo describe el proceso completo desde la detección de la necesidad hasta el despliegue de una herramienta funcional y segura.1.2 Antecedentes y motivación La motivación principal para el desarrollo de esta web nace de observar la brecha tecnológica que existe entre las aplicaciones de consumo actuales (que son rápidas, predictivas y automáticas) y los sistemas de gestión de citas médicas, que siguen siendo mayoritariamente manuales y reactivos. Como ingeniero, me resulta fascinante el reto de automatizar procesos de toma de decisiones que tradicionalmente dependen del factor humano.Lo que realmente me impulsó a elegir este tema fue la posibilidad de construir un sistema completo "de extremo a extremo". Me motiva diseñar la lógica del algoritmo de scoring, donde se decide programáticamente qué paciente es el más apto para una cita, y asegurar que esa decisión se ejecute de forma estable mediante una infraestructura web bien diseñada.A nivel personal, este proyecto representa la oportunidad de aplicar mis habilidades en desarrollo full-stack, seguridad y algoritmos para crear un producto tangible. Mi objetivo es demostrar que, mediante una buena ingeniería del software, es posible transformar un flujo de trabajo ineficiente en un proceso fluido, automático y, sobre todo, útil para la sociedad. No busco solo analizar el problema, sino construir la herramienta que lo solucione.1.3 Comparación con otras alternativasPara entender el valor de la solución desarrollada, es necesario analizar cómo se gestionan actualmente las citas médicas y qué alternativas existen en el mercado:* Sistemas de Gestión Hospitalaria (HIS) Tradicionales: La gran mayoría de centros utilizan agendas electrónicas estáticas. Su principal limitación es que son reactivas. Si un paciente cancela, el hueco aparece como libre, pero el sistema no realiza ninguna acción para ocuparlo. Dependen totalmente de que un administrativo llame por teléfono o que un nuevo paciente busque esa cita por casualidad.* Listas de Espera Manuales: Muchos centros gestionan a los pacientes mediante listas físicas o digitales estáticas. Aquí, el factor humano no solo resulta lento, sino que es ineficaz en la toma de decisiones multivariable. Un administrativo no tiene la capacidad cognitiva para calcular en segundos quién es el candidato óptimo ponderando simultáneamente tres o más criterios (urgencia, antigüedad y compatibilidad horaria). Mi motor, en cambio, resuelve este problema de optimización de forma instantánea, garantizando una objetividad que el proceso manual no puede ofrecer.* Portales de Cita Previa Online: Aunque permiten la autogestión, no resuelven el problema del no-show de última hora. Si la cancelación ocurre con poca antelación, es muy improbable que otro usuario entre al portal en ese preciso instante para ocupar el hueco.Diferenciación: A diferencia de estas alternativas, la plataforma propuesta en este TFG elimina la dependencia de la acción humana o del azar. Introduce la automatización de la demanda, donde el sistema "empuja" la cita hacia el paciente más idóneo en lugar de esperar a que el paciente encuentre el hueco.1.4 Planteamiento del sistema propuestoEl sistema diseñado se presenta como una aplicación web integral que no solo gestiona la agenda médica, sino que actúa como un agente inteligente de optimización de recursos. El planteamiento se basa en tres pilares técnicos fundamentales:1. Motor de Reasignación Proactiva: Es el núcleo del proyecto. En el momento en que se detecta una liberación de slot, el sistema inicia un proceso de búsqueda y notificación automática. No se limita a avisar a un paciente, sino que gestiona una "cascada" de ofertas para asegurar que el hueco se llene en el menor tiempo posible.2. Algoritmo de Scoring Dinámico: Para que la reasignación sea justa, el sistema utiliza un algoritmo de puntuación. No se elige a un paciente al azar, sino que se analiza quién tiene mayor urgencia clínica, quién lleva más tiempo esperando y si el horario le viene bien según su perfil. Esto garantiza una gestión ética y reduce drásticamente las listas de espera de forma automática.3. Seguridad y Verificación: El sistema protege la integridad de la atención médica mediante dos mecanismos clave:* Validación QR: Permite que cualquier receta o informe impreso sea verificado por terceros (como farmacias) para confirmar que es auténtico y no ha sido manipulado.* Resiliencia Técnica (Circuit Breaker): Garantiza que la web siga funcionando correctamente incluso ante fallos en las notificaciones o picos de tráfico, manteniendo siempre la disponibilidad para pacientes y médicos.Este planteamiento busca, en definitiva, transformar la agenda médica en un organismo vivo y optimizado, reduciendo los “tiempos muertos” y mejorando la calidad del servicio percibida por el paciente.

Alcance del Proyecto y del Producto En este capítulo se define el alcance del proyecto, detallando los productos que se entregan al finalizar el mismo, así como los factores externos y las exclusiones que condicionan el desarrollo del sistema.4.1. Entregables del proyecto Los entregables constituyen el conjunto de productos tangibles que resultan de la ejecución del TFG. Se dividen en:* E1. Producto Software: Aplicación web funcional que integra perfiles diferenciados (paciente, médico y administrador). Incluye la gestión completa (CRUD) de citas y el motor de reasignación automática de huecos libres.* E2. Memoria del TFG: Documento académico que recoge la justificación, análisis, diseño técnico, pruebas de simulación y conclusiones.* E3. Material de Presentación: Recursos visuales y documentación técnica necesaria para la defensa oral del proyecto ante el tribunal.4.2. Supuestos y restricciones del producto Los supuestos son factores que se consideran ciertos para el desarrollo del proyecto, mientras que las restricciones son limitaciones impuestas por el entorno:* S1 (Datos Simulados): Se asume que todos los datos de pacientes y médicos utilizados para la validación son ficticios, cumpliendo con la restricción de no utilizar información real.* S2 (Lógica Determinista): El algoritmo de reasignación se basa en reglas lógicas predefinidas (urgencia, antigüedad y preferencia) y no en modelos de inteligencia artificial predictiva.* S3 (Servicios de Terceros): Se asume la disponibilidad de servicios de red externos para el envío de notificaciones (servidor SMTP) y recursos de diseño, no siendo responsabilidad del software la caída de estos proveedores externos.* R1 (Plazos Académicos): El proyecto está estrictamente limitado por el calendario de la Universidad de Sevilla para la convocatoria vigente.* R2 (Prototipo Académico): El alcance se limita a un prototipo funcional para validación técnica, no siendo un producto destinado a su comercialización inmediata.4.3. 6. Diseño y Arquitectura Conceptual  Este capítulo describe el funcionamiento lógico y la estructura organizativa del sistema desde una perspectiva funcional, omitiendo detalles técnicos de implementación para centrarse en la interacción de los componentes y la lógica de negocio.6.1. Descripción funcional del ecosistema El sistema dispone de herramientas específicas para cada perfil que garantizan la continuidad asistencial y la satisfacción del usuario:* Administrador: Gestiona la infraestructura global, supervisa el estado de las agendas y configura los parámetros base del algoritmo teniendo la capacidad de ajustar la flexibilidad de los pesos según las necesidades del centro.* Médico: Actúa como el origen de la disponibilidad y gestor de la asistencia. El ecosistema garantiza la confidencialidad de los datos, permitiendo al facultativo el acceso al historial clínico digital únicamente de aquellos pacientes que ya han pasado por su consulta (Historial de Pacientes Atendidos, Ilustración 6), asegurando así una trazabilidad ética de la información médica.* Paciente: Usuario final que interactúa de forma simplificada con el sistema para gestionar sus citas y responder a las ofertas proactivas de reasignación. En su perfil privado, puede acceder y descargar sus informes y recetas, siendo obligatorio que indique su preferencia horaria (Mañana/Tarde) en su perfil, ya que es un dato que utiliza el motor para el algoritmo de reasignación.El flujo de comunicación del ecosistema se refuerza mediante un sistema de doble notificación: el paciente recibe una alerta instantánea en su panel de usuario de la aplicación y, simultáneamente, un correo electrónico automatizado (SMTP). Esta redundancia asegura que el paciente reciba el aviso de mejora de cita de forma segura, maximizando las posibilidades de que el hueco liberado sea ocupado con rapidez.El ecosistema funciona de forma reactiva: cuando se detecta un hueco libre (por cancelación), el sistema activa automáticamente el motor de búsqueda para encontrar al candidato más idóneo sin intervención manual.Es importante destacar que el sistema elimina el concepto tradicional de 'lista de espera' estática, donde el usuario debe inscribirse de forma explícita. En su lugar, el ecosistema integra una Lista de Espera Dinámica.Esta se define de forma algorítmica como el conjunto de pacientes que ya poseen una cita programada en el sistema para una fecha posterior al hueco que acaba de quedar vacante (fecha_cita > fecha_hueco_libre). De este modo, la arquitectura del software transforma la agenda futura en una cola de candidatos proactivos, permitiendo que el sistema 'empuje' las ofertas de mejora de cita a los usuarios sin que estos tengan que realizar ninguna acción adicional ni registrarse en listados externos.6.2. Lógica de priorización y sistema de pesos La principal innovación de este sistema es que sustituye la gestión manual de las agendas por un proceso automático de priorización inteligente. Mientras que en un modelo tradicional cubrir un hueco libre depende del azar o del criterio de un administrativo, este sistema utiliza un algoritmo de puntuación (scoring) que analiza objetivamente a los candidatos. Esto permite que el software decida en segundos quién es el paciente más adecuado para adelantar su cita, basándose en reglas matemáticas en lugar de decisiones subjetivas.Para garantizar una reasignación justa y eficiente, el sistema utiliza un modelo de decisión basado en una suma ponderada de factores. La puntuación de idoneidad (S) de cada candidato en lista de espera se obtiene mediante la siguiente expresión:?S=(W?_U*U)+(W_A*A)+?(W?_P*P)Para garantizar la trazabilidad y la transparencia del algoritmo, se detallan a continuación los componentes de la fórmula y los rangos en los que operan:Variables de entrada (Criterios del Candidato)* Factor de Urgencia (U): Representa el componente ético y clínico. Es un valor discreto en el rango [1,3], donde 1 corresponde a una prioridad "Baja", 2 a "Media" y 3 a "Alta". Un paciente con mayor urgencia incrementa linealmente su peso en el ranking para asegurar la atención preferente de casos graves.* Factor de Antigüedad (A): Representa el componente de justicia. Mide la distancia en días entre la fecha de la cita original del paciente y la fecha del hueco liberado (A= fecha_cita – fecha_hueco). Su rango es A?1, premiando a los pacientes que llevan más tiempo esperando para evitar el estancamiento de citas lejanas.* Factor de Preferencia (P): Evalúa la eficiencia y satisfacción del usuario según la coincidencia horaria. Se mueve en un rango discreto [-0.5,1], donde toma el valor 1 si el turno del hueco coincide con la preferencia del paciente (mañana/tarde) y -0.5 si existe una incompatibilidad horariaPesos del sistema (Configuración paramétrica)Los pesos (W) son multiplicadores configurables que permiten ajustar la sensibilidad del algoritmo:*  WU (Peso Urgencia): Multiplicador de prioridad clínica. Valor por defecto: 15.0.*  WA (Peso Antigüedad): Multiplicador de justicia temporal. Valor por defecto: 0.1.*  WP (Peso Preferencia): Multiplicador de satisfacción horaria. Valor por defecto: 10.0.Esta estructura es altamente escalable, permitiendo en el futuro incorporar nuevos factores de decisión sin alterar el núcleo del algoritmo, simplemente ajustando los pesos o añadiendo nuevos sumandos a la fórmula.6.3. Modelo de verificación externa (Validación QR)El sistema implementa un mecanismo de Integridad Documental diseñado para validar la autenticidad de los documentos emitidos (recetas o informes) fuera del ecosistema digital de la aplicación. El flujo conceptual se fundamenta en los siguientes pilares:* Generación de Identidad Digital: Al generar el documento, el sistema le asigna un identificador único (UUID) que se convierte en un código QR. Este sello garantiza que ese papel específico tiene un respaldo oficial en nuestra base de datos.* Portal de Verificación Pública (Acceso Universal): Al escanear el código, cualquier tercero interesado (como una oficina de farmacia o un centro derivado) es redirigido a un portal público que no requiere inicio de sesión. El sistema actúa como un notario digital, mostrando los datos originales de la consulta para que el tercero pueda asegurar que la información no ha sido alterada o falsificada.* Seguridad por Diseño y LOPD: Para cumplir con la normativa de protección de datos, este portal no muestra la ficha completa del paciente. Solo enseña la información mínima necesaria para verificar que el tratamiento o el diagnóstico que aparece en el papel es verídico y coincide con el registro oficial.

10. Diseño y Arquitectura Técnica  Este capítulo profundiza en la infraestructura tecnológica y los patrones de diseño avanzados que sostienen el sistema, detallando cómo la interacción de sus componentes garantiza una solución escalable, segura y eficiente.10.1. Arquitectura del software y modelo de datos El sistema adopta el patrón de arquitectura MVT (Model-View-Template), nativo del framework Django (Django Software Foundation, 2026). Esta elección permite un desacoplamiento efectivo entre la gestión de datos y la lógica de presentación.* Capa de Persistencia: El sistema utiliza una arquitectura de base de datos relacional basada en SQLite (Hipp, 2026). Esta elección técnica prioriza la portabilidad y simplicidad del prototipo, permitiendo una ejecución inmediata en cualquier entorno sin dependencias de red externas. No obstante, gracias al uso del ORM de Django (Django Software Foundation, 2026), el sistema mantiene la compatibilidad y está preparado para una migración transparente a motores de nivel empresarial si la carga asistencial lo requiriera en el futuro* Gestión de Señales (Signals): El sistema utiliza un bus de eventos interno. se dispara automáticamente una acción atómica que invoca al motor de reasignación de forma inmediata tras la detección del cambio de estado. esta integración mediante señales (signals) garantiza que la lógica de negocio esté desacoplada del controlador de la vista.10.2. Ciclo de Vida de la Reasignación y Algoritmo de ScoringEl Motor de Reasignación Inteligente (MRI) opera mediante un ciclo de vida atómico y determinista que se activa ante eventos de liberación de agenda. Este proceso garantiza que cada hueco asistencial perdido se recupere con el candidato más idóneo mediante cuatro fases:Fase 1: Disparo y Filtrado (Trigger)El proceso se inicia mediante un Signal de Django en el momento en que una cita cambia su estado a CANCELADA. El sistema identifica los metadatos del hueco (médico, fecha y turno) y realiza una consulta filtrada en la base de datos para identificar a los candidatos potenciales: pacientes con citas futuras con ese mismo facultativo (fecha_cita > fecha_hueco).Fase 2: Aplicación del Algoritmo de ScoringPara garantizar la objetividad, el motor aplica una fórmula paramétrica que pondera tres dimensiones críticas:* Preferencia de Turno (Wp): Evalúa la coincidencia entre el horario del hueco y la preferencia (Mañana/Tarde) grabada en el perfil del paciente.* Urgencia Médica (Wu): Pondera la prioridad clínica asignada a la cita original, asegurando que los casos más graves sean atendidos antes.* Antigüedad/Distancia (Wa): Premia a los pacientes que tienen su cita más lejos en el tiempo, compensando así su mayor tiempo de espera.Esta lógica de puntuación permite que el software tome una decisión basada en datos objetivos, eliminando la subjetividad del personal administrativo y optimizando la agenda en segundos. La expresión matemática que rige este cálculo es:?S=(W?_U*U)+(W_A*A)+?(W?_P*P)Donde cada peso (W) es configurable desde el panel de administración, permitiendo al centro hospitalario dar más importancia a la urgencia médica o a la antigüedad según sus necesidades operativas en tiempo real.Fase 3: Notificación y Bloqueo Temporal (TTL)El sistema selecciona al candidato con el Score más alto y genera un token único de respuesta. Se envía una notificación proactiva vía SMTP y el hueco queda en estado de "Bloqueo Temporal" durante un periodo de 24 horas (TTL). Esto evita colisiones horarias mientras el paciente evalúa la propuesta.Fase 4: Resolución o Efecto Cascada* Aceptación One-Click: Si el paciente confirma, el sistema mueve al paciente al nuevo hueco de forma atómica. Su hueco antiguo queda libre, lo que dispara una nueva ejecución del motor (Efecto Cascada) para beneficiar a un segundo paciente.* Rechazo o Expiración: Si el paciente descarta la oferta o el TTL expira, el motor descarta a dicho candidato y reinicia el proceso con el segundo mejor posicionado en el ranking de scoring.Es fundamental resaltar que este flujo es recursivo hasta un límite de seguridad definido por el Circuit Breaker. Si un paciente acepta la propuesta, su hueco original se libera, convirtiéndose en un nuevo 'disparador' que reinicia la Fase 1 para el siguiente grupo de candidatos. Este efecto cascada garantiza que una sola cancelación pueda resultar en el adelanto de múltiples citas, maximizando la eficiencia de la capacidad asistencial del centro.10.3. Mecanismos de seguridad y control de flujo (Circuit Breaker) Se ha implementado el patrón Circuit Breaker (Nygard, 2018) para monitorizar la profundidad de la recursividad en el efecto cascada. Si el sistema detecta que una sola cancelación está provocando un número inusual de reasignaciones fallidas que superan un umbral de seguridad, el "circuito" se abre. El sistema detiene la automatización para esa rama específica y deriva la gestión a una cola de revisión manual, garantizando la resiliencia del servidor.10.4. Gestión de comunicaciones y persistencia (SMTP/TTL)La capa de comunicación garantiza que la decisión del algoritmo llegue al paciente de forma efectiva:* Protocolo SMTP: La integración con servicios de correo se realiza mediante el protocolo SMTP, enviando notificaciones que contienen tokens de acceso único y temporal.* Arquitectura "One-Click": Se han diseñado Views específicas que procesan la aceptación o rechazo mediante estos tokens. Una vez que el paciente hace clic, el token queda invalidado en la base de datos.* Control de Expiración (TTL): Se implementa una lógica de Time-To-Live. Si el motor no recibe una confirmación en el tiempo parametrizado, la oferta se marca como "Expirada" y el MRI reanuda la cascada con el siguiente perfil.

Este contador rastrea de forma persistente la profundidad de la reacción en cadena, deteniendo el motor de forma atómica al alcanzar el nivel 5 de profundidad, garantizando la estabilidad global del sistema.Incidencia 2: Fallos en servicios de terceros (SMTP / Email)* El Problema: Durante las pruebas de integración, se detectó que una caída puntual o latencia alta en el servidor de correo externo bloqueaba la ejecución del motor MRI, provocando un fallo general y dejando la base de datos en un estado inconsistente.* La Solución: Se rediseñó la lógica del motor para incluir un manejo robusto de excepciones (bloques Try-Except). Actualmente, si el servicio de mensajería falla, el sistema captura el error, lo registra en un archivo de log interno y continúa con la actualización de la base de datos. Esto asegura que la disponibilidad de la agenda no dependa de la estabilidad de un proveedor de correo externo.Incidencia 3: Mantenibilidad de vistas complejas * El Problema: El crecimiento iterativo de las funcionalidades clínicas y administrativas provocó que el archivo controlador original creciera en exceso, volviéndose difícil de testear y aumentando el riesgo de introducir errores cruzados (bugs).* La Solución: Se aplicaron rigurosamente principios de Clean Code (Martin, 2008). Se refactorizó la lógica fragmentándola en una arquitectura de vistas especializadas (paciente.py, administrativo.py, clinico.py). Esta modularización fue clave para poder alcanzar el 100% de cobertura en los módulos críticos.Incidencia 4: Adaptabilidad a dispositivos móviles (Responsiveness)* El Problema: Las tablas de agendamiento médico tradicionales y el renderizado de PDFs presentaban problemas de desbordamiento visual en pantallas de teléfonos móviles, comprometiendo la accesibilidad del paciente.* La Solución: Se implementó una estrategia de diseño fluido mediante CSS Nativo (Media Queries) en el archivo premium.css. Se optimizaron los contenedores para que el flujo "One-Click" fuera intuitivo y 100% funcional desde teléfonos móvil, evitando la dependencia de frameworks pesados de terceros o la necesidad de desarrollar aplicaciones nativas (iOS/Android).

17. Anexos 17.1. Glosario de términos * AJAX (Asynchronous JavaScript and XML): Técnica de desarrollo web que permite actualizar partes de una página de forma asíncrona, enviando y recuperando datos del servidor sin necesidad de recargar la página completa. Utilizada en los filtros dinámicos del sistema.* Algoritmo de Scoring: Método lógico-matemático utilizado para asignar una puntuación numérica a distintos candidatos basándose en variables predefinidas (urgencia, antigüedad, etc.), permitiendo priorizar de forma objetiva.* Algoritmo Paramétrico: Algoritmo cuyo comportamiento se puede ajustar mediante la modificación de parámetros de entrada (como los pesos de urgencia o antigüedad) sin necesidad de reescribir el código fuente.* Auditoría (en Persistencia): Registro histórico y detallado de las operaciones críticas que permite verificar a posteriori el flujo de decisiones del sistema.* Capacidad Asistencial: Medida del volumen de servicios médicos que un centro de salud o sistema sanitario puede ofrecer en un periodo de tiempo determinado basándose en sus recursos disponibles.* Circuit Breaker (Patrón): Patrón de diseño de software utilizado para detectar fallos y evitar que un error en un módulo se propague en cascada a todo el sistema, garantizando la estabilidad del servidor.* Clean Code (Código Limpio): Conjunto de prácticas de ingeniería de software orientadas a escribir código que sea fácil de leer, entender y mantener por otros desarrolladores.* Cobertura de Código (Code Coverage): Porcentaje que indica cuánto código fuente es ejecutado por las pruebas automatizadas.* Colisión Horaria: Conflicto técnico que ocurre cuando se intenta programar dos o más citas en el mismo intervalo de tiempo para un mismo recurso (médico o sala), debiendo ser evitado por el sistema mediante transacciones seguras.* CRUD (Create, Read, Update, Delete): Acrónimo que hace referencia a las cuatro funciones básicas que debe permitir un sistema para gestionar la persistencia de datos.* Disparador (Trigger / Signal): Función o procedimiento que se ejecuta automáticamente como respuesta a un evento específico en el sistema, como la activación del motor de reasignación tras una cancelación.* Efecto Cascada: Proceso secuencial donde una acción (una reasignación) desencadena automáticamente una serie de acciones posteriores, permitiendo que un solo hueco liberado pueda beneficiar a múltiples pacientes de la lista de espera.* Escalabilidad del Modelo: Propiedad que permite al sistema crecer o añadir nuevas variables (factores de decisión) sin necesidad de rediseñar la arquitectura lógica existente.* HIS (Hospital Information System): Sistema de información integral diseñado para gestionar los aspectos operativos, administrativos y clínicos de un centro sanitario.* Hito (Milestone): Punto de referencia que marca un evento importante o la finalización de una etapa clave dentro del cronograma del proyecto.* Lógica Determinista: Enfoque de programación donde el sistema, ante una misma entrada de datos, siempre produce el mismo resultado siguiendo reglas lógicas estrictas, sin procesos aleatorios o predictivos.* LOPD: Marco normativo que regula el tratamiento de datos de carácter personal, garantizado en este sistema mediante la limitación de la exposición de datos en la validación QR.* Metodología Ágil: Enfoque de desarrollo de software basado en la entrega iterativa e incremental, permitiendo que los requisitos y soluciones evolucionen mediante el trabajo colaborativo.* No-show: Término técnico utilizado para referirse a la situación en la que un paciente con una cita programada no acude a la misma ni la cancela con antelación suficiente.* Proactivo (Software): Capacidad de un sistema informático para tomar la iniciativa y ejecutar acciones de forma autónoma ante un evento, sin esperar a que un usuario humano realice una petición manual.* RBAC (Role-Based Access Control): Modelo de seguridad que gestiona los permisos del sistema según el rol del usuario (administrativo, paciente, etc.).* Resiliencia (de software): Capacidad de un sistema informático para resistir y recuperarse rápidamente de errores, fallos de infraestructura o picos de carga, manteniendo su continuidad operativa.* Responsive: Capacidad de la interfaz web para adaptar su diseño y elementos a diferentes tamaños de pantalla y dispositivos.* Slot: Franja horaria o "hueco" específico dentro de una agenda médica destinado a la atención de un paciente.* SMTP (Simple Mail Transfer Protocol): Protocolo estándar de red utilizado para el envío y recepción de correos electrónicos.* Suma Ponderada: Método matemático que combina varios factores independientes multiplicando cada uno por un peso relativo, permitiendo obtener una puntuación final (Scoring) para la toma de decisiones.* Transacción Atómica: Conjunto de operaciones en una base de datos que se ejecutan como un bloque indivisible. Si una operación falla, todas se revierten para evitar dejar datos corruptos.* Trazabilidad Algorítmica: Capacidad de reconstruir y justificar paso a paso el proceso de decisión de un software, permitiendo auditar por qué se seleccionó un resultado específico.* Usabilidad Táctil: Optimización de los elementos de la interfaz para facilitar la interacción con los dedos en dispositivos móviles.* UUID (Universally Unique Identifier): Estándar que genera un código identificador único a nivel mundial, utilizado para securizar los códigos QR y evitar accesos por fuerza bruta.* Validación QR: Proceso de verificación de integridad de un documento mediante el escaneo de un código que enlaza con una firma o dato almacenado en una base de datos externa.* Veracidad de Datos: Principio de calidad que asegura que la información presentada en documentos externos es un reflejo exacto y actualizado de los datos almacenados en el sistema central.
